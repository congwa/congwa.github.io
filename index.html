对进程线程的理解

发表于 2019-04-18 | 更新于 2019-04-19

进程线程

任务调度主要的开销

  1. CPU执行任务调度的开销，主要是进程上下文切换的开销
  2. 任务调度后，CPU Cache/TLB不命中，导致缺页中断的开销

对于第1点的开销，不管是进程调度还是线程调度都是必须的
进程之间的内存共享，切换线程 cpu cache命中率高

##异步
实现异步的方式

  1. js中单线程语言通过事件轮询+回调函数的模式进行实现 （event loop）

##node中
js,因为 javascript是单线程的语言，所以执行必须是单进程，单线程，js的异步是通过实践轮询来实现的，任务队列的执行，通过回调函数

node中的处理并发使用异步/io模型，上层使用js，下层使用c进行多线程，，线程分配通过libuv（event loop =》所有线程同时进行运行从主线程分配，因为js单线程关系，主线程和子线程不会发生锁操作，cpu利用率高），主线程作为线程池来进行使用，事件队列，异步非io操作如:setTimeout,setInterval不会开启新线程，异步io操作 操作文件 网络请求

存钱和取钱同时并发,存钱线程会锁住资源，取钱线程阻塞。
node event loop模型:存钱线程锁住资源，给取钱线程发一个回调，取钱线程继续执行其他请求，执行完数据库拿到结果，再去执行js代码中的回调，这样 js不阻塞，多线程处理也不会遭遇阻塞，所以node.js因为event loop模型并发强大，因为js语言关系io密集型不好。

node中可以开启子进程，原因是因为js不能支持多线程，无法用js语言开启子线程

##js
js当初运行在浏览器中被设计成单线程的,因为并发操作会出现dom元素删除添加同步问题，导致出现问题不好排查。但是webkit内核本身多线程的。

##nginx服务器
nginx处理并发使用和node.js同样的 event loop模型,处理并发强大

###math

  • Math.trunc() 去除一个数的小数部分

  • Math.sign() 判断一个数到底是正数、负数、还是零。对于非数值，会先将其转换为数值。

  • Math.cbrt() 立方根

  • Math.clz32() 将参数转为 32 位无符号整数的形式，然后这个 32 位值里面有多少个前导 0。
  • Math.imul() 返回两个数以 32 位带符号整数形式相乘的结果，返回的也是一个 32 位的带符号整数。
  • Math.fround() 方法返回一个数的32位单精度浮点数形式
    对于32位单精度格式来说，数值精度是24个二进制位（1 位隐藏位与 23 位有效位），所以对于 -224 至 224 之间的整数（不含两个端点），返回结果与参数本身一致。
  • Math.hypot() 返回所有参数的平方和的平方根。

    • NFC，默认参数，表示“标准等价合成”（Normalization Form Canonical Composition），返回多个简单字符的合成字符。所谓“标准等价”指的是视觉和语义上的等价。
    • NFD，表示“标准等价分解”（Normalization Form Canonical Decomposition），即在标准等价的前提下，返回合成字符分解的多个简单字符。

    • NFKD，表示“兼容等价分解”（Normalization Form Compatibility Decomposition），即在兼容等价的前提下，返回合成字符分解的多个简单字符。

sql约束

###NOT NULL

  • NOT NULL 约束强制列不接受 NULL 值。
  • NOT NULL 约束强制字段始终包含值。这意味着，如果不向字段添加值，就无法插入新记录或者更新记录。

###UNIQUE
UNIQUE 约束唯一标识数据库表中的每条记录。

  • UNIQUE 和 PRIMARY KEY 约束均为列或列集合提供了唯一性的保证。
  • PRIMARY KEY 拥有自动定义的 UNIQUE 约束。
  • 请注意，每个表可以有多个 UNIQUE 约束，但是每个表只能有一个 PRIMARY KEY 约束。

    DEFAULT

    DEFAULT 约束用于向列中插入默认值。
    如果没有规定其他的值，那么会将默认值添加到所有的新记录。

###CHECK
CHECK 约束用于限制列中的值的范围。

  • 如果对单个列定义 CHECK 约束，那么该列只允许特定的值。
  • 如果对一个表定义 CHECK 约束，那么此约束会在特定的列中对值进行限制。

###FOREIGN KEY (外键)
一个表中的 FOREIGN KEY 指向另一个表中的 PRIMARY KEY。

  • FOREIGN KEY 约束用于预防破坏表之间连接的动作。
  • FOREIGN KEY 约束也能防止非法数据插入外键列，因为它必须是它指向的那个表中的值之一。

PRIMARY KEY (主键)

PRIMARY KEY 约束唯一标识数据库表中的每条记录。

  • 主键必须包含唯一的值。
  • 主键列不能包含 NULL 值。
  • 每个表都应该有一个主键，并且每个表只能有一个主键。

###CONSTRAINT
约束用于限制加入表的数据的类型。

可以在创建表时规定约束（通过 CREATE TABLE 语句），或者在表创建之后也可以（通过 ALTER TABLE 语句）。

以下几种约束：

NOT NULL
UNIQUE
PRIMARY KEY
FOREIGN KEY
CHECK
DEFAULT

同源策略

浏览器的同源策略是 协议 域名 端口号

  • cookie
  1. cookie的同源策略 仅仅只限域名
  2. 每次通讯服务器都会发送一个cookie，保存在cookie默认路径下
  3. 经过测试,cookie在同域共享，留意cookie覆盖情况。
  4. cookie的访问路径默认为用户访问路径
  5. 客户端向服务器通讯时，携带cookie(未禁用),cookie的会匹配父路径的所有cookie一同发送
  6. 受不同浏览器url编码不同限制，cookie存储数据应该进行编码，不然有可能会导致中文乱码情况
  7. 大小一般4kb
  • sessionStorage
  1. 浏览器同源策略下唯一
  2. 每个浏览器签页唯一 页签关闭就删除
  3. 一般最大5mb
  • sessionStorage
  1. 没有时间限制 存在硬盘中
  2. 相同浏览器的不同页面间可以共享相同域名和端口相同的 localStorage

###登陆验证
目前通常服务器登陆验证操作通过两种方式
1:jwt
2:cookie中保存了sessionId 通过sessionId去服务器取出session信息

一个请求对应一个cookie,cookie中携带sessionid,每次请求服务器都可以(也可以选择不创建)创建一个session和生成一个sessionid，用户随意记录信息，session保存在服务器内存中,在服务器关闭或者到期后释放或者用户自己释放，通过sessionid进行获取。（不同服务器的session实现方式稍微不一样）

###csrf
一次用户点击导致的惨案 -> 非常详细的例子
在网站使用jwt、sessionId的验证的方案的时候，如果用户有效的防范了xss攻击，非常难突破进行csrf攻击。(欢迎指正)

###xss
在网站输入框写入可执行代码进行盗取用户登录信息

xss攻击主要分为两种方式
  • 在url注入可执行代码
  • 通过表单输入框提交注入可执行代码
防御方法主要通过白名单检测进行防御(待续)
  • 防御富文本后端 jsoup
  • 前端npm包 xss cssfilter
  • koa-helmet
